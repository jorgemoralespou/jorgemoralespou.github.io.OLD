    Test Microsoft Draft on minishift

    8 minutes Posted: 19/Jul/2017 by Jorge Morales

    After experimenting with helm charts, it is time to experiment with Microsoft Draft.

    In their own words:

    > Draft makes it easy to build applications that run on Kubernetes. Draft targets the “inner loop” of a developer’s workflow: as they hack on code, but before code is committed to version control. > > Using Draft is as simple as: > > 1. draft create to containerize your application based on Draft packs > 2. draft up to deploy your application to a Kubernetes dev sandbox, accessible via a public URL > 3. Use a local editor to modify the application, with changes deployed to Kubernetes in seconds > > Once the developer is happy with changes made via Draft, they commit and push to version control, after which a continuous integration (CI) system takes over. Draft builds upon [Kubernetes Helm][helm] and the Kubernetes Chart format, making it easy to construct CI pipelines from Draft-enabled applications.

    So it is my time to experiment with it, and provide you a guide to follow so you can also experiment with it. I hope this guide will make your journey easier.

    Install

    I’ve been following the guide here to install draft, but I have adapted the steps to make it work on minishift. All the instructions have been tested on my local laptop, which is a mac, but should be easy to follow on other Operating Systems, specially on any Linux based distro.

    If you have already followed the instructions on my previous blog about helm, there is quite some repeated stuff, as helm is a dependency and requirement for draft. If you have not done that already, please go to that other blog and follow the instructions there. I will resume from the point where you have helm installed on your laptop and helm tiller deployed on minishift.

    Start minishift

    If you don’t have minishift up and running, it is time to start it up. As the memory, cpu and hypervisor configuration used when creating minishift are preserved, you just need to do start.

    $ minishift start --vm-driver=virtualbox --openshift-version=v3.6.0-rc.0 --cpus=2 --memory=8192
Starting local OpenShift cluster using 'virtualbox' hypervisor...
Starting OpenShift using openshift/origin:v3.6.0-rc.0 ...
OpenShift server started.

The server is accessible via web console at:
    https://192.168.99.100:8443

    NOTE: Remember to use the same flags you used before, as minishift don’t remember them for now.

    Install additional requirements. A docker registry.

    One of the requirements that Draft has is a Docker registry to push the images we build. As OpenShift has one registry, we will expose the registry to the world via an Openshift Route. For this, minishift provides an addon (not yet merged as it requires OpenShift >= 3.6), so let’s install and enable it.

    $ cd /tmp
$ git clone https://github.com/minishift/minishift
$ cd minishift/addons
$ minishift addons install registry-route
$ minishift addons apply registry-route
-- Applying addon 'registry-route':........
Add-on 'registry-route' created docker-registry route. Please run following commands to login to the OpenShift docker registry:
  eval $(minishift docker-env)
  docker login -u developer -p `oc whoami -t` docker-registry-default.192.168.99.100.nip.io:443
$ eval $(minishift docker-env)
$ docker login -u developer -p `oc whoami -t` docker-registry-default.192.168.99.100.nip.io:443

    Install Draft (Client and server):

    It is time to install draft. You can find the latest releases of draft for your Operating System in their Install instructions GitHub page. Grab the appropriate binary for you, unpack and add it to your path.

    Whenever we install Draft, it will fail, mostly because it requires to run a privileged container. In a later step I will fix it by creating a dedicated serviceaccount for draft and allowing this serviceaccount to run privileged containers.

    Why I haven’t granted the SCC to the default serviceaccount? Mainly because doing so would alter the behavior of everything else deployed in the same project.

    As it happened with helm, I need to provide draft with the –kube-context and –host flags. You can go back to my previous blog to see how to get those.

    $ export TILLER_HOST="$(minishift ip):$(oc get svc/tiller -o jsonpath='{.spec.ports[0].nodePort}' -n kube-system --as=system:admin)"

$ draft init --kube-context default/192-168-99-100:8443/system:admin --host $TILLER_HOST

    NOTE: You will need to execute draft as privileged, so I will create a serviceaccount named draft in the kube-system namespace and will grant the provileged SCC to it.

    $ oc create serviceaccount draft -n kube-system --as=system:admin
$ oc adm policy add-scc-to-user privileged system:serviceaccount:kube-system:draft --as=system:admin

    I will verify that the deployment has actually failed. Then I will patch the deployment to use the draft serviceaccount, and will redeploy draftd.

    $ oc get deployment/draftd -n kube-system --as=system:admin

$ oc patch deployment/draftd -p '{"spec":{"template":{"spec":{"serviceAccountName":"draft"}}}}' -n kube-system --as=system:admin
deployment "draftd" patched

$ oc get deployment/draftd -o yaml -n kube-system --as=system:admin | grep serviceAccount
      serviceAccount: draft
      serviceAccountName: draft

$ oc get deployment/draftd -n kube-system --as=system:admin
NAME      DESIRED   CURRENT   UP-TO-DATE   AVAILABLE   AGE
draftd    1         1         1            1           3m

    Draft creates a folder in $HOME/.draft where it will store the packs and plugins.

    Now we really have draftd up and running!

    Currently it looks like draft needs to open a tunnel to your server. In order to do that, the only way I have been able to make it work has been to port-forward my local port 44135 to the draftd pod’s port 44135.

    $ oc port-forward $(oc get pod xxxx -n kube-system --as=system:admin) :44135 -n kube-system --as=system:admin &amp;

    Now we have draftd listening locally on port 44135.

     Getting started

    Once we have Draft up and running on OpenShift (minishift), we will install our first application. For sake of simplicity, our first application will use the Getting Started guide on Draft site.

    First thing we will need is to get the example applications on my local laptop:

    $ cd /tmp
$ git clone https://github.com/Azure/draft
$ cd draft/examples/python

    This guide uses a simple Python application which uses Flask to provide a very simple Hello World webserver.

    Draft Create

    We need some “scaffolding” to deploy our app into a Kubernetes cluster. Draft can create a Helm chart, a Dockerfile and a draft.toml with draft create:

    $ draft create
--&gt; Python app detected
--&gt; Ready to sail
$ ls -a
.draftignore  Dockerfile  app.py  chart/  draft.toml  requirements.txt

    The chart/ and Dockerfile assets created by Draft default to a basic Python configuration. This Dockerfile harnesses the python:onbuild image, which will install the dependencies in requirements.txt and copy the current directory into /usr/src/app. And to align with the service values in chart/values.yaml, this Dockerfile exposes port 80 from the container.

    The draft.toml file contains basic configuration about the application like the name, which namespace it will be deployed to, and whether to deploy the app automatically when local files change.

    $ cat draft.toml
[environments]
  [environments.development]
    name = "tufted-lamb"
    namespace = "default"
    watch = true
    watch_delay = 2

    Edit this file to adapt to our configuration. Change the namespace to myproject.

    $ cat draft.toml
[environments]
  [environments.development]
    name = "my-pythonapp"
    namespace = "myproject"
    wait = false
    watch = false
    watch_delay = 2

    See the Draft User Guide for more information and available configuration on the draft.toml.

    A .draftignore file is created as well for elements we want to exclude tracking on draft up when watching for changes. The syntax is identical to helm ignore feature.

    Note that for technical reasons, the .git/ directory is unconditionally ignored for now.

    Draft Up

    Now we’re ready to deploy app.py to a Kubernetes cluster.

    Draft handles these tasks with one draft up command:

    • reads configuration from draft.toml
    • compresses the chart/ directory and the application directory as two separate tarballs
    • uploads the tarballs to draftd, the server-side component
    • draftd then builds the docker image and pushes the image to a registry
    • draftd instructs helm to install the Helm chart, referencing the Docker registry image just built

    Interact with the Deployed App

    Using the handy output that follows successful deployment, we can now contact our app.

    $ curl http://tufted-lamb.example.com
Hello Draft!

    When we curl our app, we see our app in action! A beautiful “Hello Draft!” greets us. If not, make sure you’ve followed the Ingress Guide.

    Update the App

    Now, let’s change the output in app.py to output “Hello Kubernetes!” instead:

    $ cat &lt;<eof> app.py
from flask import Flask

app = Flask(__name__)

@app.route("/")
def hello():
    return "Hello Kubernetes!\n"

if __name__ == "__main__":
    app.run(host='0.0.0.0', port=8080)
EOF
